# Regime Terminal

An HMM-driven regime detection and backtesting system. Trains a Hidden Markov
Model on historical hourly market data, auto-labels the resulting hidden
states as market regimes (e.g. `crash` → `chop` → `strong_bull`), overlays a
configurable technical-confirmation strategy on top of those regimes,
backtests it with realistic risk controls (leverage, cooldowns, stop-losses),
and serves both a CLI and a Streamlit dashboard for exploration.

> ⚠️ **This is educational/research software, not financial advice.** See
> [Caveats](#caveats--limitations) before using this with real capital.

---

## How it works

```
dataloader.py      -> fetch & cache hourly OHLCV (yfinance, default BTC-USD)
features.py        -> returns, intraday range, volume change, rolling vol
hmmmodel.py         -> fit a Gaussian HMM (hmmlearn, or sklearn GMM fallback)
regimelabeler.py    -> auto-label HMM states by ranking mean return
strategies.py       -> 8 technical confirmations (RSI, MACD, ADX, etc.)
backtester.py       -> sequential bar-by-bar simulation with risk rules
dashboard.py        -> Streamlit UI tying it all together
cli.py              -> run a backtest or get a live signal from the terminal
utils.py            -> metrics (Sharpe/Sortino/drawdown), plotting helpers
```

## Quickstart

### Option A: Google Colab

1. Push this repo to GitHub (see [Getting this onto GitHub](#getting-this-onto-github) below) or upload the files directly to a Colab session.
2. Open `notebooks/regime_terminal_colab.ipynb` in Colab.
3. Run the cells top to bottom. The notebook clones/installs dependencies, loads BTC-USD hourly data for the last 730 days, fits the HMM, runs the backtest, and shows all charts inline.

### Option B: Local / VS Code

```bash
git clone https://github.com/<your-username>/regime-terminal.git
cd regime-terminal
python -m venv .venv && source .venv/bin/activate   # or your preferred env manager
pip install -r requirements.txt

# Run a backtest from the terminal:
python cli.py --run-backtest --asset BTC-USD --leverage 2.5 --confirmations 7

# Get the current live signal:
python cli.py --live-signal --asset BTC-USD

# Launch the interactive dashboard:
streamlit run dashboard.py
```

### Running tests

```bash
pip install pytest
pytest tests/ -v
```

All tests use synthetic, deterministic data (no network access required), so
they run the same in CI as they do locally.

## Configuration

All parameters live in `config.py` as a single `Config` dataclass. Every
module reads from a `Config` instance, so you can override anything from
Python, the CLI, or the dashboard sidebar without touching multiple files:

```python
from config import Config
cfg = Config(asset="ETH-USD", leverage=3.0, confirmations_required=6)
```

### Default hyperparameters

| Parameter | Default | Description |
|---|---|---|
| `asset` | `BTC-USD` | Ticker passed to yfinance |
| `interval` | `1h` | Candle interval |
| `lookback_days` | `730` | History window (yfinance caps hourly data at ~730 days) |
| `n_components` | `7` | Number of HMM hidden states |
| `hmm_backend` | `hmmlearn` | `hmmlearn` (true HMM) or `gmm` (sklearn fallback) |
| `random_state` | `42` | Seed for reproducibility |
| `confirmations_required` | `7` | Of 8 total confirmations, how many must agree to enter |
| `leverage` | `2.5` | Position leverage multiplier |
| `cooldown_hours` | `48` | Hours to wait after an exit before re-entering |
| `min_hold_hours` | `24` | Minimum hours to hold before a confirmation-based exit is allowed |
| `trade_cost` | `0.0005` | Commission + slippage, charged per side (round-trip = 2x) |
| `stop_loss_pct` | `0.08` | Hard stop-loss, 8% adverse move from entry |
| `trailing_stop_pct` | `0.04` | Trailing stop distance from peak (if enabled) |

### Aggressive mode

```python
cfg = Config().apply_aggressive()
# leverage -> 4.0x, confirmations_required -> 5/8, trailing stop on at 1.5%
```

In the dashboard, this is a single sidebar checkbox. Aggressive mode trades
more often and with more leverage — it is **not** a strictly "better" setting,
just a different risk profile.

## The 8 confirmation checks

Entries require `confirmations_required` (default 7) of these 8 to be true,
**in addition to** the HMM regime being bullish:

1. **RSI** — not in overbought territory
2. **Momentum (ROC)** — rate-of-change is positive
3. **ADX** — trend strength above threshold (filters out chop)
4. **MACD** — MACD line above signal line
5. **Volatility filter** — realized vol isn't in an extreme spike
6. **Volume spike** — volume above its rolling mean by N std devs
7. **Price action** — breakout above the recent rolling high
8. **Moving average** — price above its rolling MA

All thresholds (`rsi_bull_max`, `adx_threshold`, `vol_spike_std`, `ma_period`,
etc.) are configurable in `config.py`.

## Regime auto-labeling

The HMM produces arbitrary, unordered state IDs (0 through 6 by default) —
which ID maps to "bull" vs. "crash" is not fixed and can shuffle between
retrains. `regimelabeler.py` solves this by ranking states on **mean return**
after fitting: highest mean → `strong_bull`, lowest → `crash`, with
`bear` / `weak_bear` / `chop` / `weak_bull` / `bull` in between. This mapping
is stored alongside the model so it stays consistent until the next retrain.

## Outputs

Running a backtest (via CLI, notebook, or dashboard) gives you:

- **Regime-labeled price chart** — close price with background shading per regime
- **Posterior probability heatmap** — HMM confidence per state, per bar
- **Equity curve** — strategy vs. buy-and-hold
- **Drawdown chart**
- **Trade log** — entry/exit time & price, leverage, PnL, exit reason (exportable to CSV)
- **Metrics** — total return, alpha vs. buy-and-hold, Sharpe, Sortino, max drawdown, win rate
- **Per-state performance** — how trades entered in each regime actually performed

The dashboard additionally includes a few features aimed at people new to
trading terminology, all in `explain.py`, `regimelabeler.py`, and
`prediction_log.py`:

- **Plain-English signal explanation** — translates the regime/confidence/
  confirmation numbers into a sentence describing what's happening and why
  (`explain_signal`).
- **Historical transition panel** — "looking back at every time the market
  was in regime X, what did the *next* period turn out to be, historically."
  This is explicitly framed as a historical pattern, not a forecast — the
  language in `explain_transition` is deliberately hedged and tested to
  avoid implying certainty (see `tests/test_explain.py`).
- **Stop-loss / trailing-stop price levels** — shows the actual dollar price
  at which the backtester's rules would exit if you bought right now. Comes
  with an explicit caveat that this only executes automatically *inside the
  simulation* — on a real or demo exchange, you need to place an actual
  stop-loss/stop-limit order yourself for this protection to apply.
- **Signal log + hindsight grading** (`prediction_log.py`) — every dashboard
  run automatically saves a snapshot (regime, confidence, confirmations,
  price, and the top historical "most likely next regime" guess) to
  `signal_log.csv`. Once enough bars have passed for the outcome to be
  knowable, the dashboard checks whether that guess was actually right and
  shows a running accuracy score. This file is gitignored by default since
  it's personal usage history, not project code.

## Retraining

```python
from cli import run_pipeline
from config import Config

cfg = Config(asset="BTC-USD")
pipeline = run_pipeline(cfg, force_refresh=True)  # bypasses the data cache
pipeline["model"].save(f"models/regime_hmm_{cfg.asset}.joblib")
```

A reasonable cadence is weekly or after every ~500 new candles — frequent
enough to adapt to regime changes, infrequent enough to avoid overfitting to
noise. There's no automatic scheduler built in; wire this into a cron job,
GitHub Action, or your orchestration tool of choice.

## Caveats & Limitations

- **Overfitting risk.** This system fits the HMM and backtests on the same
  historical window by default. Use a walk-forward split (fit on an earlier
  period, test on a later one, roll forward) before trusting any backtest
  number. Cross-validate `n_components` and `confirmations_required` rather
  than hand-picking values that look good on one run.
- **Backtest ≠ live performance.** The backtester assumes fills at the close
  price with a flat `trade_cost`. Real execution has latency, partial fills,
  and slippage that scales with order size and volatility.
- **yfinance hourly data is capped at ~730 days** by the API itself — this is
  a Yahoo Finance limitation, not a bug in `dataloader.py`.
- **Regime labels are descriptive, not predictive.** "Bull" means this state
  historically had the highest mean return *in the fitted window* — it is not
  a guarantee about what happens next.
- **This is not financial advice.** Leveraged trading, especially in crypto,
  carries substantial risk of loss. Consult a licensed financial advisor
  before trading with real capital.
